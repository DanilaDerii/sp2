SP2 Project Log
================

Date: 2026-04-17

Purpose
-------
SP2 is being designed as a local-first course AI assistant platform.
The product direction is:

- one reusable local app
- downloadable course-specific packs/modules
- teacher-side preprocessing for heavy document ingestion
- student-side lightweight local runtime for asking questions against installed course packs


High-Level Product Direction
----------------------------
- Students should install one app, not one app per course.
- Each course should be represented as a modular downloadable course pack.
- Course packs should contain preprocessed course knowledge so students do not need to parse raw PDFs locally.
- Retrieval should be isolated per course by default.
- The app should support local LLM usage and local retrieval without hosted APIs.
- The v1 product direction is functionality-first, not platform-first.
- SP2 should complement systems like MS Teams rather than replace them.


Locked Stack So Far
-------------------
Backend:
- Python
- FastAPI

Frontend:
- TypeScript
- frontend app not scaffolded yet

Metadata storage:
- SQLite

Vector store:
- LanceDB

Audio transcription:
- faster-whisper

Document parsing / OCR:
- Docling
- intended mainly for authoring / teacher-side preprocessing

Local model runtime:
- Ollama
- Ollama is treated as an external machine-level runtime, not a pip dependency

Embedding direction:
- one shared embedding model should be used for both:
  - teacher-side chunk embeddings
  - student-side question embeddings
- embedding model should be fixed for v1 to avoid retrieval mismatch


Architecture Decisions Discussed
--------------------------------
1. Course packaging model
- preferred approach is one app plus many downloadable course packs
- not one separate app per course
- professor exports AI-ready pack
- student imports AI-ready pack into local app

2. Separation of responsibilities
- teacher/build pipeline can use heavy tooling like Docling
- student/runtime side should stay lighter
- heavy separation is deferred for now, but we should keep the ingestion boundary clean

3. Parsing distinction
- Docling is for course material parsing
- student questions do not need heavy parsing
- student questions only need lightweight preprocessing, embedding, retrieval, and answer generation

4. Retrieval constraint
- chunk embeddings and student-question embeddings must come from the same embedding model family/version
- otherwise vector retrieval quality will degrade or fail

5. Database direction
- we decided not to use Postgres + pgvector for SP2
- rationale: local packaging and local-first setup are better served by SQLite + LanceDB

6. Product scope refinement
- we do NOT want to build a full course platform in v1
- professor should be able to use the app as a local pack builder
- professor will likely gather source materials from an existing platform such as MS Teams
- professor shares the built pack back through normal channels such as Teams / Drive / similar
- student app consumes the supplementary AI pack in parallel with the existing LMS/course directory

7. Feedback loops
- feedback / analytics loop for professor is intentionally deferred for now
- current focus is:
  - professor builds pack
  - student imports pack
  - student asks questions locally

8. Pack content decision
- course pack should NOT contain raw source materials such as textbooks, PDFs, or lecture recordings
- course pack should contain only AI-ready derived data from those sources
- this includes things like:
  - extracted/normalized text
  - chunks
  - chunk metadata
  - embeddings
  - retrieval-ready/index-ready data

9. Pack format direction
- pack should be file-based, not a portable Postgres-style database
- likely export format:
  - structured folder zipped into a custom extension such as `.sp2pack`
- app runtime may still choose to load imported pack data into SQLite/LanceDB locally
- distinction:
  - pack format = portable file archive
  - runtime state/index = local installed representation

10. Revised minimal data model
- simplified v1 pack files:
  - `pack.json`
  - `chunks.json`
  - `vectors.npy`
- pack should contain only AI-ready derived data, not raw source materials
- runtime SQLite entities reduced to:
  - `InstalledPack`
  - `ChatSession`
  - `ChatMessage`
- runtime `ChatMessage.role` means chat turn type:
  - `user`
  - `assistant`
- no separate persisted citation/source table for v1
- runtime retrieval corpus should be represented locally in LanceDB
- detailed revision captured in `DATA_MODEL_LOG.txt`

11. V1 pack build/import workflow
- professor-side builder is responsible for the heavy pipeline
- professor provides source materials such as:
  - textbook PDFs
  - lecture slides/PDFs
  - optional supplementary lecture videos
- builder processes sources sequentially for easier debugging in v1
- pipeline outputs only derived AI-ready data:
  - extracted/normalized text
  - chunks
  - chunk metadata
  - embeddings
- exported v1 pack remains:
  - `pack.json`
  - `chunks.json`
  - `vectors.npy`
- exported format for v1 should be plain `.zip`
- exported zip is supplementary course data only and must NOT include original PDFs, textbooks, slides, or videos
- professor shares the zip through normal course channels such as Teams
- student app imports the zip, validates pack compatibility, and creates the local runtime state from it

12. V1 retrieval/runtime decision
- embedding model should be fixed for v1 across both build side and student runtime side
- this avoids retrieval mismatch and makes debugging simpler
- chunk metadata should preserve referencing helpers when available, including:
  - source title
  - source type
  - chunk index
  - page or page range for PDF-derived chunks
  - section / topic / subtopic when parsing can recover it
  - timestamp or timestamp range for video-derived transcript chunks
- LanceDB should NOT be exported as part of the pack
- instead, the student app should read `chunks.json` + `vectors.npy` on import and populate the local LanceDB index then
- SQLite remains responsible for installed pack metadata and chat history

13. V1 assistant behavior direction
- assistant behavior should stay minimal in v1 so we can focus on making the retrieval pipeline work first
- tutor style, direct-answer style, and assignment-specific behavior can be refined later once the system is working end to end
- for early debugging, the assistant should explicitly say when the answer is not in the installed pack / retrieval database
- this makes retrieval failures and pack-coverage gaps easier to spot during development

14. V1 source granularity / referencing direction
- source granularity should primarily support clean referencing rather than an overly complex source model in v1
- for now, each imported source file should be treated as a source item
- chunk metadata should preserve enough structure to reference:
  - source file / source title
  - topic / subtopic when parsing can recover it
  - exact page or page range for document-derived chunks
- this should be enough for early citations and navigation without overdesigning the first schema

15. V1 import/update semantics
- pack update/replacement semantics are intentionally deferred for v1
- we do not need to fully decide yet how duplicate imports, newer versions of the same `pack_id`, or multi-version coexistence should behave
- for now, the priority is getting single-pack import, retrieval, and chat working end to end

Important Notes Learned
-----------------------
- `requirements.txt` cannot lock the Python interpreter version by itself
- Python version is effectively locked by:
  - `.python-version`
  - creating the venv from the intended interpreter
- venv rebuild is needed when changing Python version
- venv rebuild is NOT needed when only adding more packages later

- On this Linux environment, Docling pulls a very heavy dependency tree through torch
- because of that, splitting runtime and heavy document requirements is the right move

Prototype App Plan
==================

Date: 2026-04-24

Goal
----
Build one local-first web app that contains both the teacher workflow and the student workflow.

The app should act as:
- a teacher-side pack builder
- a student-side pack importer/runtime
- a local AI chat interface over imported course-pack data

This is currently the preferred prototype direction because it lets us prove the full product loop inside one app without prematurely splitting the system into multiple applications.


High-Level UX Direction
-----------------------
- landing page should immediately present two modes:
  - teacher POV
  - student POV
- both modes live inside the same web application
- all RAG, AI, and database components remain local to the machine
- app should not rely on hosted APIs for the core prototype workflow


Teacher POV
-----------
Purpose:
- let a professor prepare AI-ready course packs from source materials

Expected user flow:
1. Teacher opens the teacher page from the landing page
2. Teacher selects source files from local storage
3. Selected files may include:
   - textbook PDFs
   - lecture slides / lecture PDFs
   - optional supplementary lecture videos
4. Teacher starts the build pipeline
5. The app processes the selected files locally
6. The app outputs the derived pack files
7. The app compresses the pack into a downloadable `.zip`
8. Teacher downloads that zip and shares it through the normal course channel such as Teams

Teacher-side pipeline responsibilities:
- parse source materials
- extract text
- normalize/clean extracted text
- chunk the text
- attach chunk metadata
- generate embeddings using the fixed v1 embedding model
- write pack artifacts for export

Teacher-side output:
- `pack.json`
- `chunks.json`
- `vectors.npy`
- compressed into plain `.zip` for v1

Important teacher-side constraints:
- exported zip must contain only derived AI-ready data
- exported zip must NOT contain original course materials such as PDFs, textbooks, slides, or videos
- build pipeline should run sequentially in v1 to keep debugging simpler


Student POV
-----------
Purpose:
- let a student import an exported pack and use it locally for retrieval and chat

Expected user flow:
1. Student opens the student page from the landing page
2. Student clicks import and selects a course-pack `.zip`
3. The app imports the pack into local runtime storage
4. The app validates the pack metadata and compatibility
5. The app creates the required local databases / indexes from the imported files
6. The imported pack becomes available in the student-side UI
7. Student selects the pack and navigates to chat

Student-side import responsibilities:
- unpack the zip into local installed-pack storage
- read `pack.json`
- read `chunks.json`
- read `vectors.npy`
- populate SQLite app/runtime metadata
- populate local LanceDB retrieval index from chunk + vector data

Important student-side constraints:
- LanceDB should be created locally during student import
- LanceDB should not be shipped as part of the exported pack
- SQLite should store installed-pack metadata and chat history


Chat / Retrieval Flow
---------------------
Purpose:
- allow the student to ask questions against the imported course-pack data

Expected runtime flow:
1. Student chooses an installed pack
2. Student opens the AI chat page for that pack
3. Student submits a question
4. The app embeds the question using the same fixed v1 embedding model
5. LanceDB retrieves the closest chunks for that question
6. Retrieved chunks are passed into the answer-generation step as context
7. The assistant responds using that retrieved context

Debug-first behavior for v1:
- assistant behavior should remain simple for now
- if the answer is not present in the installed pack / retrieval database, the assistant should explicitly say so
- this is primarily for debugging retrieval quality and pack coverage during early development


Information Architecture
------------------------
Preferred early page structure:

1. Landing page
- choose teacher POV or student POV

2. Teacher page
- file selection
- list of selected materials
- build/start action
- progress/status display
- download exported zip

3. Student page
- import zip action
- list of installed packs
- select pack
- navigate to chat

4. Chat page
- ask questions against selected installed pack
- show retrieved references where possible
- show explicit debug message when answer is not in the database


Why This Prototype Shape Is Preferred
-------------------------------------
- one app is simpler to build and demo than multiple separate apps
- teacher and student workflows remain clearly separated in UI while sharing one local runtime
- it validates the whole course-pack lifecycle end to end:
  - build
  - export
  - import
  - index
  - chat
- it stays aligned with the local-first direction already decided for SP2


Recommended Build Order
-----------------------
1. Scaffold the frontend shell for landing page, teacher page, student page, and chat page
2. Add the teacher-side source selection and build UI
3. Add the student-side zip import UI
4. Implement import-time creation/population of SQLite + LanceDB
5. Implement the chat/retrieval path over imported pack data
6. Replace temporary/stubbed internals with the full parsing/chunking/embedding pipeline as needed

This order is meant to get a working prototype shell visible early while still converging toward the intended architecture.
