SP2 Project Idea
================

SP2 is a local-first course AI assistant built around a teacher-to-student
course pack workflow.

The core idea is simple: teachers prepare course-specific AI packs from course
materials, and students import those packs through the SP2 student app or an LM
Studio extension/plugin. The import seeds the local SP2 database, then students
ask questions through LM Studio's model manager and chat UI. The goal is not to
replace an LMS, Teams, Drive, normal course folders, or LM Studio. SP2 is meant
to sit beside those tools as a focused retrieval layer that grounds LM Studio
answers in the actual course content.


Problem
-------
Students often receive course knowledge through many scattered sources:
textbooks, PDFs, lecture slides, recordings, announcements, Teams or LMS posts,
and personal notes. When they have a question, they may not know which file,
page, lecture, or topic contains the answer.

Generic AI assistants can explain concepts, but they are not naturally scoped to
one course. They may answer from general model knowledge instead of the material
the teacher actually assigned. This is risky for studying because a confident
answer can still be unsupported, incomplete, or inconsistent with the class.

Cloud document assistants improve grounding, but they still require account
access, uploads, usage limits, and per-student setup. They also move course
materials and student questions into a cloud workflow, which can create privacy,
copyright, and consistency concerns.

Teachers have a related problem. They want students to get useful help, but they
do not want to manually answer the same questions repeatedly, lose control over
what material the assistant uses, or ask every student to build their own
inconsistent AI setup. A practical system should let the teacher prepare the
course knowledge once and distribute it through the same channels they already
use.


Product Vision
--------------
SP2 should provide one reusable local app that supports many downloadable course
packs.

On the teacher side, the application gathers selected course materials and runs
them through a RAG build pipeline. The pipeline extracts text, normalizes it,
splits it into searchable chunks, attaches useful metadata, creates embeddings,
and exports a portable pack zip.

On the student side, the application or LM Studio extension/plugin imports that
pack and seeds local storage. Students then use LM Studio as the main interface
for model selection, chat, and answer generation. SP2 provides the custom
retrieval script/tool behind that interface: it embeds the question, searches the
seeded local database first, returns matching course chunks when available, and
lets the LM Studio language model compose the final answer from that context.
The student should be able to ask natural questions, continue a conversation
across messages, and navigate back to the course material through references
such as source title, page, section, topic, or timestamp when available.

The assistant should feel course-aware rather than generic. It should preserve
the context of student and AI communication, retrieve from the selected course
pack first, and produce answers grounded in the local course database when
matching material exists. If no relevant course material is found in the
database, the question may fall back to the LM Studio model's pretrained
knowledge instead of pretending the answer came from the pack.


Teacher Workflow
----------------
The teacher workflow is responsible for preparing the course pack.

Expected flow:

1. The teacher collects source material from existing course channels.
2. The teacher selects a PDF, a bundle of files, or selected files from a bundle.
3. SP2 runs the selected sources through the teacher-side RAG pipeline.
4. The pipeline extracts text and source structure where possible.
5. The pipeline chunks the material and stores metadata for later references.
6. The pipeline creates embeddings with the fixed course-pack embedding model.
7. SP2 writes the pack files and exports them as a zip.
8. The teacher shares the exported zip through Teams, LMS, Drive, or another
   normal distribution channel.

The exported pack is derived data only. It must not include the original raw
course materials such as PDFs, textbooks, slides, videos, or recordings.

The current v1 pack format is:

- `pack.json`
- `chunks.json`
- `vectors.npy`

`pack.json` stores pack metadata and compatibility information. `chunks.json`
stores extracted chunk text and chunk metadata. `vectors.npy` stores the
embedding matrix, with vector order matching chunk order.


Student Workflow
----------------
The student workflow is responsible for local pack installation, retrieval, and
LM Studio integration.

Expected flow:

1. The student receives a course pack zip from the teacher.
2. The student imports the pack into the SP2 app or LM Studio extension/plugin.
3. SP2 validates the pack files and metadata.
4. SP2 stores installed-pack metadata in SQLite.
5. SP2 reads `chunks.json` and `vectors.npy`.
6. SP2 creates or updates the local LanceDB retrieval index.
7. The student uses LM Studio as the main chat and local model interface.
8. The student asks questions while the SP2 tool checks the selected pack first.
9. SP2 embeds the question, retrieves relevant chunks, and sends the retrieved
   context to the LM Studio language model for answer composition.
10. If no relevant chunks are found, the question can be answered from the LM
   Studio model's pretrained knowledge.
11. SP2 may store conversation messages locally when the student-side runtime
   needs continuity outside LM Studio's own chat history.

The student app or LM Studio integration should be privacy-first. Course-pack
import, retrieval, and local database state should stay on the student device
for the core workflow. The system should not require uploading student
questions, chat history, or installed course material to a hosted AI service.


LM Studio Runtime
-----------------
SP2 is now designed to use LM Studio as the main student-facing local model
service/server. LM Studio should provide the student's model manager, model
selection, chat UI, and language model generation. SP2 should provide the course
pack builder, pack importer, local database seeding, vector retrieval, and the
custom script/tool that LM Studio calls before answering.

The embedding model must match between teacher-side pack creation and
student-side question embedding. If the teacher embeds chunks with one embedding
model and the student embeds questions with a different incompatible model,
retrieval quality will degrade or fail. For v1, the embedding model should stay
fixed to keep the system predictable.

The generation model should be swappable through LM Studio based on the
student's hardware. A student with a weaker laptop should be able to use a
smaller model, while a student with stronger hardware can choose a larger or
better model. This model choice should affect answer generation, not the pack
compatibility contract.

The current direction is:

- local embedding model for retrieval compatibility
- LM Studio as the main local generation service and chat UI
- SP2 retrieval tool/script called before answers are generated
- database-first answering when relevant chunks exist
- pretrained-model fallback when the seeded database has no relevant material
- support future model selection based on available hardware
- avoid hosted APIs for the core prototype workflow


Storage Model
-------------
SP2 separates portable pack data from installed runtime data.

The pack zip is the teacher-to-student distribution contract. It is portable and
file-based.

The installed student runtime state is local to the student app:

- SQLite stores installed pack metadata, threads, and messages.
- LanceDB stores searchable chunk text, chunk metadata, and vectors.

This keeps the exported pack simple while allowing the student app to build its
own local retrieval index during import.


What Makes SP2 Different
------------------------
SP2 is not a generic chatbot. It is a course-pack-based local assistant.

Important differences:

- Teacher prepares one shared course pack.
- Students import the same course-grounded pack.
- Retrieval is isolated to the selected installed course pack.
- The exported pack contains derived AI-ready data, not raw source files.
- Student questions, pack data, and retrieval state stay local.
- Answers should be grounded in course material and should expose references
  when relevant pack chunks are found.
- If the seeded database has no relevant material, LM Studio can answer from the
  selected model's pretrained knowledge.
- The app complements existing course platforms instead of replacing them.
- LM Studio model selection can adapt to different student hardware.


Current Development Direction
-----------------------------
The current implementation is moving toward separate teacher and student
workflows inside the same overall project.

The teacher side is currently ahead. It has a PDF-first pipeline that can
extract course content with Docling, chunk the extracted text, embed chunks
through Ollama, write pack files, and export a zip.

The student side is being rebuilt around the pack contract and LM Studio
integration. Its core storage shape is SQLite for installed packs, threads, and
messages, plus LanceDB for pack chunks and vectors. The next major milestone is
to implement pack import, database seeding, retrieval, and an LM Studio
tool/plugin path that searches the seeded database before the model answers.

Future teacher-side expansion should add more source types:

- PDFs
- slides / PowerPoint
- audio or video transcripts
- mixed course-material bundles

Future response behavior can include tutor-style or course-specific answer
styles, but those settings should only be added to the pack contract once the
student runtime can actually honor them.


Short Summary
-------------
SP2 lets teachers convert course materials into portable local AI packs.
Students import those packs into a privacy-first local app or LM Studio
extension/plugin, seed a local database, and ask questions through LM Studio.
The system is designed to retrieve source-aware course context first, let LM
Studio compose answers from that context, and fall back to the selected model's
pretrained knowledge only when the database has no relevant material.
