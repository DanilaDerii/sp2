SP2 Project Log
================

Date: 2026-04-17

Purpose
-------
SP2 is being designed as a local-first course AI assistant platform.
The product direction is:

- one reusable teacher/student pack workflow
- downloadable course-specific packs/modules
- teacher-side preprocessing for heavy document ingestion
- student-side lightweight local runtime or LM Studio extension/plugin for
  importing packs and searching installed course databases
- LM Studio as the main student-facing model server, model manager, and chat UI


High-Level Product Direction
----------------------------
- Students should install one app, not one app per course.
- Each course should be represented as a modular downloadable course pack.
- Course packs should contain preprocessed course knowledge so students do not need to parse raw PDFs locally.
- Retrieval should be isolated per course by default.
- The app should support local retrieval and LM Studio-based local LLM usage
  without hosted APIs.
- The v1 product direction is functionality-first, not platform-first.
- SP2 should complement systems like MS Teams rather than replace them.


Locked Stack So Far
-------------------
Backend:
- Python
- FastAPI

Frontend:
- TypeScript
- frontend app not scaffolded yet

Metadata storage:
- SQLite

Vector store:
- LanceDB

Audio transcription:
- faster-whisper

Document parsing / OCR:
- Docling
- intended mainly for authoring / teacher-side preprocessing

Local model runtime:
- LM Studio is now the target main service/server for the student runtime.
- LM Studio should own local model management, chat UI, and answer generation.
- SP2 should own pack creation, pack import, database seeding, and retrieval.
- Ollama remains part of the existing teacher-side prototype implementation
  until that code path is replaced or bridged.

Embedding direction:
- one shared embedding model should be used for both:
  - teacher-side chunk embeddings
  - student-side question embeddings
- embedding model should be fixed for v1 to avoid retrieval mismatch


Architecture Decisions Discussed
--------------------------------
1. Course packaging model
- preferred approach is one app plus many downloadable course packs
- not one separate app per course
- professor exports AI-ready pack
- student imports AI-ready pack into the SP2 app or LM Studio extension/plugin

2. Separation of responsibilities
- teacher/build pipeline can use heavy tooling like Docling
- student/runtime side should stay lighter
- heavy separation is deferred for now, but we should keep the ingestion boundary clean

3. Parsing distinction
- Docling is for course material parsing
- student questions do not need heavy parsing
- student questions only need lightweight preprocessing, embedding, retrieval,
  and context handoff to LM Studio for answer generation

4. Retrieval constraint
- chunk embeddings and student-question embeddings must come from the same embedding model family/version
- otherwise vector retrieval quality will degrade or fail

5. Database direction
- we decided not to use Postgres + pgvector for SP2
- rationale: local packaging and local-first setup are better served by SQLite + LanceDB

6. Product scope refinement
- we do NOT want to build a full course platform in v1
- professor should be able to use the app as a local pack builder
- professor will likely gather source materials from an existing platform such as MS Teams
- professor shares the built pack back through normal channels such as Teams / Drive / similar
- student app or LM Studio extension/plugin consumes the supplementary AI pack
  in parallel with the existing LMS/course directory

7. Feedback loops
- feedback / analytics loop for professor is intentionally deferred for now
- current focus is:
  - professor builds pack
  - student imports pack
  - student asks questions through LM Studio with SP2 retrieval first

8. Pack content decision
- course pack should NOT contain raw source materials such as textbooks, PDFs, or lecture recordings
- course pack should contain only AI-ready derived data from those sources
- this includes things like:
  - extracted/normalized text
  - chunks
  - chunk metadata
  - embeddings
  - retrieval-ready/index-ready data

9. Pack format direction
- pack should be file-based, not a portable Postgres-style database
- likely export format:
  - structured folder zipped into a custom extension such as `.sp2pack`
- app runtime may still choose to load imported pack data into SQLite/LanceDB locally
- distinction:
  - pack format = portable file archive
  - runtime state/index = local installed representation

10. Revised minimal data model
- simplified v1 pack files:
  - `pack.json`
  - `chunks.json`
  - `vectors.npy`
- pack should contain only AI-ready derived data, not raw source materials
- runtime SQLite entities reduced to:
  - `InstalledPack`
  - `Thread`
  - `ChatMessage`
- runtime `ChatMessage.role` means message author type:
  - `user`
  - `assistant`
- no separate persisted citation/source table for v1
- runtime retrieval corpus should be represented locally in LanceDB
- detailed revision captured in `DATA_MODEL_LOG.txt`

11. V1 pack build/import workflow
- professor-side builder is responsible for the heavy pipeline
- professor provides source materials such as:
  - textbook PDFs
  - lecture slides/PDFs
  - optional supplementary lecture videos
- builder processes sources sequentially for easier debugging in v1
- pipeline outputs only derived AI-ready data:
  - extracted/normalized text
  - chunks
  - chunk metadata
  - embeddings
- exported v1 pack remains:
  - `pack.json`
  - `chunks.json`
  - `vectors.npy`
- exported format for v1 should be plain `.zip`
- exported zip is supplementary course data only and must NOT include original PDFs, textbooks, slides, or videos
- professor shares the zip through normal course channels such as Teams
- student app or LM Studio extension/plugin imports the zip, validates pack
  compatibility, and creates the local runtime state from it

12. V1 retrieval/runtime decision
- embedding model should be fixed for v1 across both build side and student runtime side
- this avoids retrieval mismatch and makes debugging simpler
- chunk metadata should preserve referencing helpers when available, including:
  - source title
  - source type
  - chunk index
  - page or page range for PDF-derived chunks
  - section / topic / subtopic when parsing can recover it
  - timestamp or timestamp range for video-derived transcript chunks
- LanceDB should NOT be exported as part of the pack
- instead, the student app or LM Studio extension/plugin should read
  `chunks.json` + `vectors.npy` on import and populate the local LanceDB index
  then
- SQLite remains responsible for installed pack metadata and chat history
- LM Studio is the preferred student-facing chat/model layer; SP2 should expose
  a retrieval tool/script that LM Studio can call before composing an answer

13. V1 assistant behavior direction
- assistant behavior should stay minimal in v1 so we can focus on making the retrieval pipeline work first
- tutor style, direct-answer style, and assignment-specific behavior can be refined later once the system is working end to end
- default behavior should be database-first: retrieve relevant pack chunks and
  use them as context for the LM Studio model's answer
- if no relevant pack chunks are found, the question may fall back to the LM
  Studio model's pretrained knowledge
- during debugging, the system should still expose whether an answer used pack
  retrieval or pretrained-model fallback

14. V1 source granularity / referencing direction
- source granularity should primarily support clean referencing rather than an overly complex source model in v1
- for now, each imported source file should be treated as a source item
- chunk metadata should preserve enough structure to reference:
  - source file / source title
  - topic / subtopic when parsing can recover it
  - exact page or page range for document-derived chunks
- this should be enough for early citations and navigation without overdesigning the first schema

15. V1 import/update semantics
- pack update/replacement semantics are intentionally deferred for v1
- we do not need to fully decide yet how duplicate imports, newer versions of the same `pack_id`, or multi-version coexistence should behave
- for now, the priority is getting single-pack import, retrieval, and chat working end to end

Prototype App Plan
==================

Date: 2026-04-24

Goal
----
Build a local-first prototype that contains the teacher pack builder and the
student pack import/retrieval workflow, with LM Studio as the preferred
student-facing chat and model layer.

The app should act as:
- a teacher-side pack builder
- a student-side pack importer/runtime or LM Studio extension/plugin
- a local retrieval layer that feeds LM Studio before answer generation

This direction still proves the full product loop, but the student-facing UI
should preferably be LM Studio instead of a custom chat screen when the plugin
or tool integration path is available.


High-Level UX Direction
-----------------------
- landing page should immediately present two modes:
  - teacher POV
  - student POV
- both modes live inside the same web application
- all RAG, AI, and database components remain local to the machine
- LM Studio provides the main student chat UI and local LLM manager
- app should not rely on hosted APIs for the core prototype workflow


Teacher POV
-----------
Purpose:
- let a professor prepare AI-ready course packs from source materials

Expected user flow:
1. Teacher opens the teacher page from the landing page
2. Teacher selects source files from local storage
3. Selected files may include:
   - textbook PDFs
   - lecture slides / lecture PDFs
   - optional supplementary lecture videos
4. Teacher starts the build pipeline
5. The app processes the selected files locally
6. The app outputs the derived pack files
7. The app compresses the pack into a downloadable `.zip`
8. Teacher downloads that zip and shares it through the normal course channel such as Teams

Teacher-side pipeline responsibilities:
- parse source materials
- extract text
- normalize/clean extracted text
- chunk the text
- attach chunk metadata
- generate embeddings using the fixed v1 embedding model
- write pack artifacts for export

Teacher-side output:
- `pack.json`
- `chunks.json`
- `vectors.npy`
- compressed into plain `.zip` for v1

Important teacher-side constraints:
- exported zip must contain only derived AI-ready data
- exported zip must NOT contain original course materials such as PDFs, textbooks, slides, or videos
- build pipeline should run sequentially in v1 to keep debugging simpler


Student POV
-----------
Purpose:
- let a student import an exported pack, seed the local SP2 database, and use LM
  Studio for chat over retrieved course context

Expected user flow:
1. Student opens the student page from the landing page
2. Student clicks import in the SP2 app or LM Studio extension/plugin and
   selects a course-pack `.zip`
3. SP2 imports the pack into local runtime storage
4. The app validates the pack metadata and compatibility
5. The app creates the required local databases / indexes from the imported files
6. The imported pack becomes available to the student-side runtime/tool
7. Student asks questions in LM Studio, with SP2 retrieval running first

Student-side import responsibilities:
- unpack the zip into local installed-pack storage
- read `pack.json`
- read `chunks.json`
- read `vectors.npy`
- populate SQLite app/runtime metadata
- populate local LanceDB retrieval index from chunk + vector data

Important student-side constraints:
- LanceDB should be created locally during student import
- LanceDB should not be shipped as part of the exported pack
- SQLite should store installed-pack metadata and chat history


Chat / Retrieval Flow
---------------------
Purpose:
- allow the student to ask questions in LM Studio while SP2 checks the imported
  course-pack data first

Expected runtime flow:
1. Student chooses or configures an installed pack for the LM Studio session
2. Student asks a question in LM Studio
3. The SP2 tool/script receives the question before answer generation
4. SP2 embeds the question using the same fixed v1 embedding model
5. LanceDB retrieves the closest chunks for that question
6. If relevant chunks are found, SP2 passes them into LM Studio as answer context
7. LM Studio's selected language model composes the answer from that context
8. If no relevant chunks are found, LM Studio can answer from the selected
   model's pretrained knowledge

Debug-first behavior for v1:
- assistant behavior should remain simple for now
- the system should clearly track whether an answer used retrieved pack chunks
  or pretrained-model fallback
- this is primarily for debugging retrieval quality and pack coverage during
  early development


Information Architecture
------------------------
Preferred early page structure:

1. Landing page
- choose teacher POV or student POV

2. Teacher page
- file selection
- list of selected materials
- build/start action
- progress/status display
- download exported zip

3. Student page
- import zip action
- list of installed packs
- select pack
- navigate to chat

4. LM Studio chat/session
- ask questions through LM Studio with a selected installed pack configured
- show retrieved references where possible
- show whether the answer used retrieved pack chunks or pretrained-model
  fallback


Why This Prototype Shape Is Preferred
-------------------------------------
- one app plus LM Studio integration is simpler to demo than building a full
  custom student chat platform
- teacher and student workflows remain clearly separated while the student chat
  surface can be handled by LM Studio
- it validates the whole course-pack lifecycle end to end:
  - build
  - export
  - import
  - index
  - chat
- it stays aligned with the local-first direction already decided for SP2


Recommended Build Order
-----------------------
1. Scaffold only the SP2 UI needed for teacher pack build and student pack import
2. Add the teacher-side source selection and build UI
3. Add the student-side zip import UI or LM Studio extension/plugin import path
4. Implement import-time creation/population of SQLite + LanceDB
5. Implement the LM Studio retrieval tool/script over imported pack data
6. Replace temporary/stubbed internals with the full parsing/chunking/embedding pipeline as needed

This order is meant to get a working prototype shell visible early while still converging toward the intended architecture.

LM Studio Student Runtime Direction Update
==========================================

Date: 2026-06-23

Decision
--------
- Keep the teacher-side pack builder concept mostly the same.
- Use LM Studio as the main student-facing local model service/server.
- Student chat should happen through LM Studio's platform where possible,
  including its local LLM manager, model selection, and chat UI.
- SP2 should provide the student-side import/retrieval layer as an app,
  extension, plugin, or local tool script callable by LM Studio.
- Importing a teacher pack should seed the local SP2 runtime database:
  - SQLite for installed pack metadata and app/runtime state
  - LanceDB for vector-searchable course chunks
- Student questions should search the seeded SP2 database before using the
  language model's general pretrained knowledge.

Runtime Flow
------------
1. Teacher creates a course pack in the SP2 teacher app.
2. Student installs the SP2 student app or LM Studio extension/plugin.
3. Student imports the pack.
4. SP2 validates the pack and seeds SQLite/LanceDB.
5. Student asks a question in LM Studio.
6. The SP2 tool/script embeds the question and searches LanceDB first.
7. If relevant chunks are found, SP2 returns those chunks and source metadata to
   LM Studio.
8. LM Studio's selected local language model composes the answer using that
   retrieved course context.
9. If no relevant chunks are found, the question can be answered from the
   selected model's pretrained knowledge.

Implication
-----------
- The old custom student chat UI is no longer the default target.
- Build only the student UI that LM Studio cannot provide, mainly pack import,
  pack management, debug/status, and retrieval configuration.
- Debug output should distinguish retrieval-grounded answers from pretrained
  fallback answers.

Teacher Pack Metadata Simplification
====================================

Date: 2026-06-06

Decision
--------
- Removed the v1 pack metadata field `tutor_mode` from the teacher-side export contract.
- Keep pack metadata focused on retrieval/runtime compatibility fields for now:
  - `pack_id`
  - `title`
  - `version`
  - `description`
  - `embedding_model`
  - `embedding_dim`
  - `default_top_k`
  - `created_at`
  - `builder_version`


Rationale
---------
- Response style per pack is still a valid future idea, but it is not implemented in the current student runtime.
- Keeping it in `pack.json` now would make the export contract imply behavior the app does not actually support.
- Future work can reintroduce a tutor/style mode once the response-generation layer is ready to honor it.


Teacher Chunk Section Heuristic
===============================

Date: 2026-06-06

Decision
--------
- Added a v1 `section` heuristic for teacher-generated chunks.
- Each page's first meaningful extracted text line is stored as the chunk `section` for chunks from that page.
- `topic` remains unset for now.


Rationale
---------
- Slide-style PDFs usually expose the slide title as the first meaningful page line, which is useful for student-side source context.
- This deterministic heuristic avoids adding another model/classification step before the retrieval pipeline is stable.
- A better future implementation should use Docling structure where possible and may use an LLM or classifier to assign richer `section` and `topic` metadata for improved student responses.
