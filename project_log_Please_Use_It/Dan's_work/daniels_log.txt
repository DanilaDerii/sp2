24.06.26 - locked student answer architecture for v1: LM Studio owns the student chat UI and final answer generation, while SP2 owns pack import, question embedding, LanceDB retrieval, and returning a structured context packet. SP2 should not default to being a separate custom chat app or the main answer generator. The core tool behavior should be `sp2_get_course_context`: if relevant course chunks are found, return `mode = course_context` with chunk text and source metadata; if no useful chunks are found, return `mode = no_course_context` with an explicit no-course-context message. LM Studio then decides the final wording in its own chat, including any generic fallback answer, while clearly separating course-grounded answers from model-general answers.

24.06.26 - future-proofing note: v1 still uses a fixed retrieval embedding model (`all-minilm:latest`) so teacher-built chunk vectors and student question vectors stay compatible. Added a separate student retrieval embedding-model policy module so this rule is not hard-coded throughout the retrieval pipeline. Later, SP2 should support fully dynamic embedding model selection on both teacher and student sides: the teacher pack builder should record the selected embedding model and dimension in `pack.json`, and the student runtime should load/select the matching local query embedder before retrieval. Until that is implemented, mismatched embedding models should fail clearly instead of silently returning bad search results.

24.06.26 - future retrieval quality note: current v1 retrieval is already semantic in the basic embedding sense because teacher chunks and student questions are converted into vectors and searched by vector similarity instead of simple keyword matching. Later, improve semantic quality across both teacher and student sides by testing stronger local embedding models, improving teacher-side chunking so chunks hold cleaner concepts, adding hybrid vector + keyword search for exact terms, adding reranking after initial LanceDB retrieval, and optionally rewriting student questions into cleaner search queries before embedding. Keep the v1 path simple first: compatible embeddings, LanceDB top-k vector search, then context packet decision.

24.06.26 - checkpoint: implemented the first student FastAPI layer for pack management and retrieval context. Added `student/backend/app.py`, `api/routes/packs.py`, and `api/routes/retrieval.py`. Pack routes expose installed-pack listing, single-pack lookup, and local `.zip` import by path while delegating actual unpacking/seeding to the existing importer. Retrieval route exposes `POST /retrieval/context`, which accepts `installed_pack_id`, `question`, optional `top_k`, and optional `max_distance`, then returns either `course_context` with retrieved chunk text/source metadata/distance/score or `no_course_context` with an explicit no-context message. Verified the API on `127.0.0.1:8001`: `/health`, `/packs`, `/packs/5`, and `/retrieval/context` all returned expected data for the installed Week02 pack. This is now the local HTTP boundary that a later LM Studio tool/script can call.

24.06.26 - checkpoint: implemented and tested the direct student `.zip` pack importer. 
The importer now accepts the teacher-exported zip without requiring manual unpacking, 
validates the v1 pack contract (`pack.json`, `chunks.json`, `vectors.npy`), 
extracts the pack into `student/storage/installed_packs/{pack_id}-{version}/`, 
creates the SQLite `installed_packs` row, and seeds LanceDB `pack_chunks` with matching chunk text, 
metadata, and vectors. Tested with `artifacts/week02---unit-testing.zip`; 
SQLite installed pack id `5` matched the extracted `pack.json`, 
LanceDB contained `28` rows for that installed pack, 
and the first checked chunk rows matched the artifact metadata and corresponding 384-dim vectors from `vectors.npy`. 
Also added cleanup behavior for failed imports and kept `installed_packs/` 
in git with `.gitkeep` while ignoring imported pack contents.
